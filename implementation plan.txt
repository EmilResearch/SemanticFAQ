
# Implementation Plan: FAQ Retrieval System (Q-Q Matching Advanced)

## 1. Overview del Progetto
Sviluppare un sistema di Information Retrieval basato su Q-Q Matching (Query-to-Query). Il sistema NON usa LLM in fase di risposta per evitare allucinazioni, velocizzare l'esecuzione e azzerare i costi di runtime. 
In fase di ingestion, Gemini estrae coppie Q&A con varianti semantiche. A runtime, la query dell'utente viene embeddata, vengono recuperati i Top-10 match e gli score vengono **aggregati per `answer_id`** per trovare la risposta più robusta, azzerando il problema della similarità ambigua. Quindi il top-10 match serve per capire se la domanda dell'utente recupera piu volte la stessa risposta e nel caso quella che recupera piu volte la stessa risposta, ha un punteggio maggiore.


## 2. Tech Stack & Prerequisiti
- **Linguaggio:** Python 3.9+
- **LLM (solo per Ingestion & Judge):** Google Gemini `gemini-2.5-flash` tramite libreria `litellm`.
- **Modello Embedding:** Google Gemini `text-embedding-004` (dimensione vettore: 768) tramite libreria `litellm`.
- **Vector Database:** Qdrant in modalità locale (libreria `qdrant-client` con path locale, no Docker richiesto).
- **Document Store:** Un semplice file `data.json` che fungerà da database relazionale per le risposte.
- **Parsers:** `pypdf` per i PDF, lettura standard per `.txt` e `.md`.
- **Variabili d'ambiente:** La chiave API (`GEMINI_API_KEY`) deve essere letta da file `.env` usando `python-dotenv`.
- **Configurazione:** Un file `config.py` centralizzerà tutte le costanti per evitare "magic numbers".

## 3. Struttura del Progetto
```text
project_root/
├── source/                 # Cartella con i documenti originali (pdf, txt, md)
├── failed_chunks/          # Salvataggio automatico dei chunk se il parsing JSON fallisce (per debug)
├── config.py               # Costanti centralizzate (modelli, soglie, dimensioni)
├── data.json               # Output dello script 1 (Document Store)
├── prepare_data.py         # Script 1: Generazione Q&A (con LLM-as-a-Judge opzionale)
├── ingest_to_qdrant.py     # Script 2: Embedding in batch e popolamento Vector DB
├── chatbot.py              # Script 3: Interfaccia CLI con Top-K Aggregation
├── missed_queries.log      # Log automatico delle query utente fallite (sotto soglia)
├── requirements.txt        # Dipendenze
└── .env                    # Chiavi API
```

## 4. Dettagli Implementativi degli Script

### File: `config.py`
Centralizza la configurazione:
- `LLM_MODEL = "gemini/gemini-2.5-flash"`
- `EMBEDDING_MODEL = "gemini/text-embedding-004"`
- `VECTOR_SIZE = 768`
- `QDRANT_PATH = "./qdrant_data"`
- `COLLECTION_NAME = "faq_matching"`
- `MAX_TOKENS_DOC = 10000`, `CHUNK_SIZE = 8000`, `OVERLAP = 500`
- `TOP_K_RETRIEVAL = 10`
- `DEFAULT_THRESHOLD = 0.70`

### Script 1: `prepare_data.py` (Generazione Dati)
**Obiettivo:** Leggere i file in `source/`, processarli con Gemini per estrarre coppie Domanda/Risposta (con `source_quote` e `category`) e salvarle in `data.json`.
**Specifiche:**
1. Lettura file: Scorre ricorsivamente `source/`. Estrai il testo da PDF, TXT, MD.
2. **Token Counting Corretto:** Stima i token usando `litellm.token_counter(model="gemini/gemini-2.5-flash", text=...)`. NON usare tiktoken.
3. **Chunking Condizionale:** Se il documento ha <= 10.000 token, invialo intero. Altrimenti, dividilo in chunk da 8.000 token con overlap di 500.
4. **LLM Call (JSON Mode):** Invia il testo a Gemini usando il prompt fornito (Sezione 5). Forza l'output in JSON passando `response_format={"type": "json_object"}` a `litellm.completion`.
5. **Parsing Robusto:** Se il JSON restituito è malformato o manca di chiavi obbligatorie, salva il chunk testuale in `failed_chunks/{filename}_{chunk_idx}.txt` e continua col prossimo senza fermarti.
6. Assegnazione ID: Per ogni blocco valido, genera un `id` (UUID4). Salva l'oggetto come `{"id": "...", "risposta": "...", "domande": ["q1", "q2", "q3"], "category": "...", "source_file": "...", "source_quote": "..."}`.
7. **LLM-as-a-Judge (OPZIONALE):** Abilitabile con `--judge`. Se attivo, per ogni blocco invia a Gemini il `source_quote` e la `risposta` usando l'apposito prompt (Sezione 5). Se il verdetto è "NO", scarta il blocco e logga un warning.
8. Salvataggio: Scrivi l'array finale in `data.json` (sovrascrivi se esiste).

### Script 2: `ingest_to_qdrant.py` (Vector DB Ingestion)
**Obiettivo:** Leggere `data.json`, generare gli embedding in batch per tutte le domande e salvarli in Qdrant locale.
**Specifiche:**
1. Inizializza `QdrantClient(path="./qdrant_data")`.
2. **Idempotenza:** Usa `client.recreate_collection()` ad ogni avvio per ripartire da zero ed evitare duplicati nascosti.
3. Leggi `data.json`. Estrai *tutte* le domande di *tutti* i blocchi in un'unica lista, mantenendo traccia del `block_id` e `category` per ogni domanda.
4. **Batch Embedding:** Invia le domande a `litellm.embedding` in batch da 100 (es. `input=[lista_di_100_domande]`). Questo ridurrà i tempi di ingestion del 90%. Mappa i vettori restituiti alle domande originali tramite l'indice della lista.
5. Upsert su Qdrant: Inserisci ogni embedding come record separato.
   - Payload: `{"answer_id": "<id_blocco>", "category": "<categoria>", "question_text": "<testo_domanda>", "source_file": "<file_origine>"}`.

### Script 3: `chatbot.py` (CLI Chatbot con Top-K Aggregation)
**Obiettivo:** Loop CLI per interagire con il sistema. Nessun LLM usato qui. Alta precisione garantita dall'aggregazione.
**Specifiche:**
1. Carica `data.json` in memoria come dizionario `answers_dict = {item["id"]: item for item in data}`.
2. Loop `while True`. Accetta un argomento CLI `--threshold` (default: 0.70).
3. Quando l'utente inserisce una query:
   a. Genera l'embedding della query dell'utente usando Gemini.
   b. Cerca in Qdrant `limit=TOP_K_RETRIEVAL` (es. 10 risultati).
   c. **Aggregazione Score:** Itera sui 10 risultati. Crea un dizionario `scores_by_answer_id`. Per ogni risultato, somma lo score (`result.score`) all'`answer_id` corrispondente nel dizionario. (Se 3 domande varianti matchano la stessa risposta, il loro score si sommerà, decretando il vincitore assoluto).
   d. Trova l'`answer_id` con lo score aggregato più alto.
   e. **Controllo Soglia (OOD):** Se lo score aggregato Top-1 è inferiore alla `threshold`, stampa "Non ho trovato informazioni..." e **logga la query su `missed_queries.log`** (in formato append, con query, score e timestamp).
   f. Se supera la soglia, recupera il blocco da `answers_dict[answer_id]`.
   g. Stampa la risposta a terminale. Stampa in fondo anche la fonte: `Fonte: {source_file}`.

## 5. Prompt per Gemini (Script 1)

**Prompt di Generazione Q&A:**
```text
Sei un sistema esperto di Information Retrieval e Data Ingestion per motori di ricerca semantici. 

Il tuo obiettivo è convertire i documenti forniti in un set di blocchi informativi indipendenti. 

Rispetti rigorosamente le seguenti linee guida:
1. COPERTURA TOTALE: Non tralasciare alcun dettaglio, dato tecnico, procedura o eccezione presente nel testo. Tutto il valore informativo deve essere estratto.
2. STRUTTURA DEL BLOCCO: Ogni blocco deve contenere:
   - "risposta": Un testo esaustivo, chiaro e autocontenuto che spiega un concetto o una procedura specifica.
   - "domande": Una lista di 3-5 varianti di domande che trovano risposta *esattamente* e *solo* in quel testo (varia stile: formale, colloquiale, keyword-focused).
   - "category": Una categoria breve per il blocco (es. "Check-in", "Elettrodomestici", "Pagamenti").
   - "source_quote": La frase o paragrafo ESATTO dal quale hai estratto la risposta (copiato letteralmente).
3. ISOLAMENTO: Le risposte non devono richiedere la lettura del resto del documento.
4. NON INVENTARE: Estrai solo le informazioni esplicitamente scritte. Se il testo è vago, ignoralo.

Genera l'output ESCLUSIVAMENTE in formato JSON (un oggetto con chiave "blocks" contenente l'array dei blocchi):
{
  "blocks": [
    {
      "risposta": "Testo completo...",
      "domande": ["Domanda 1?", "Domanda 2?", "Domanda 3?"],
      "category": "Nome Categoria",
      "source_quote": "Testo letterale dal documento..."
    }
  ]
}

Ecco il contenuto da processare:
[INSERIRE_QUI_IL_TESTO_DEL_CHUNK]
```

**Prompt per LLM-as-a-Judge (se --judge attivo):**
```text
Sei un sistema di verifica incrociata. Ti viene fornito un estratto di documento (Source Quote) e una risposta generata (Risposta).
Verifica se la Risposta è supportata al 100% dalla Source Quote, senza aggiunte o inferenze esterne.
Rispondi ESCLUSIVAMENTE in formato JSON:
{
  "verdetto": "SI" o "NO",
  "motivazione": "Breve spiegazione"
}

Source Quote:
{source_quote}

Risposta:
{risposta}
```

## 6. Gestione degli Errori e Best Practices
- Aggiungi un meccanismo di `retry` con backoff esponenziale su `litellm` (es. `num_retries=3`) per gestire i rate limits di Gemini.
- Se un chunk fallisce il parsing JSON, non fermare la pipeline: salvalo in `failed_chunks/` per ispezionarlo manualmente in un secondo momento.
- Mantieni i log puliti usando `print()` strutturati per l'utente, ma usa il file `missed_queries.log` per tracciare le prestazioni del retrieval nel tempo.
